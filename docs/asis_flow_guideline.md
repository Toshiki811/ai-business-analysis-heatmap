# As-Isフロー作成ガイドライン

実務の現場でも使えるAs-Isフローを描くための評価軸と作図ルール。`prompts/04_asis_detail_prompt.md`（フロー生成）と `scripts/verify_outputs.mjs`（機械検証。分岐健全性は `scripts/lib/flows.mjs` の `checkFlowStructure`）はこの文書を基準にする。

---

## §0 目的（評価の基準点）

As-Isフローは目的によって正しい描き方が変わる。本ツールの目的は次の一点に固定する:

> **AI・自動化による改善余地の特定。網羅性より可読性を優先し、人手の「判断・転記・照合・ハンドオフ」がどこにどれだけあるかが一目で読めること。**

この基準から、以下すべての粒度・分岐・例外の判断が導かれる。「起こりうる例外を全部描く」ことは目的に対してノイズであり、避ける。

---

## §1 良いフローの6観点（レビュー用チェックリスト）

現場の担当者が見て「これはウチの業務だ」と言える条件。各項目に検証手段を併記する。

| # | 観点 | 判定基準 | 検証 |
|---|------|---------|------|
| ① | 正常系が1本の主線で読める | start→endを最頻ルートで指でなぞれる | 目視 |
| ② | 粒度の縦串が揃う | 隣接ノードの抽象度が同じ／§2の分割トリガー準拠 | 半自動（ノード数・動詞数） |
| ③ | ハンドオフが見える | 主体が変わる点でレーンをまたぐ（改善のボトルネックは大抵ここ） | 目視＋半自動 |
| ④ | 手作業/システム/判断の境界で割れている | AI代替を検討する単位になっている | 目視 |
| ⑤ | **分岐が構造として正しい** | decisionは出口2本以上・2経路以上・各枝にラベル・conditionあり（§⑤） | **自動**（`scripts/verify_outputs.mjs`） |
| ⑥ | システム外作業が明示されている | Excel・紙・口頭などの非システム作業が見える（実態との乖離ポイント） | 目視 |

## §2 粒度ルール（カウントではなく分割トリガーで決める）

ノード数の絶対値（「○ノード以上」）は指標として弱く、数合わせで中身が雑になりやすい。**何が起きたらノードを割るか／割らないか**で決める。

### 割る（＝別ノードにする）トリガー
1. 主体が変わる（ハンドオフ）
2. 使うシステム・ツールが変わる（会計システム→Excel→紙）
3. 待ち・承認・受け渡しが発生する
4. 判断（経路が分岐する）が入る
5. 媒体が変わる（紙起票→システム入力）

### 割らない（＝1ノードにまとめる）条件
- 同一主体が同一ツールで連続して行い、途中に判断もハンドオフもない作業はまとめる。過分割もまた可読性を損なう。

抽象工程名（「受付業務」「起案」等）は禁止。「請求書と納品書の照合」のように**対象と動作**が分かる名称にする。ノード数は目安として「業務種別あたり概ね10〜25、6以下は粗すぎ」を残すが、上記トリガーが優先。

## §3 差戻し・分岐ポリシー

「入力1つごとに差戻し確認が刺さる」問題は、検証を逐次に置くことから生じる。設計原則を入れ替える。

1. **検証はゲートに集約する。** 入力ごとにチェックを挟まず、一連の入力をまとめた後に1つのレビュー/承認ゲートを置き、不備は一括で差戻す。「早期に弾かれれば後段不要」はゲート集約で構造的に解消する。
2. **差戻しは状態が後退するものだけ描く。** その場で直して進む「自己修正」は分岐にせず、ノードの説明文に書く。フロー上の差戻しエッジ（`edge_type:"return"`）にするのは **(a) 他主体へ差し戻す（レーンをまたぐ）** か **(b) 前工程へ戻る** もののみ。
3. **decisionは経路が実際に変わる分岐だけ。** 95%通過する品質チェックは分岐に値しない。結果が同じなら描かない。
4. **例外は影響度×頻度で上位のみ明示ノード化する。** 残りはフロー化せず `unmodeled_notes` に理由とともに退避。網羅しない。ただし**影響度の高い例外（金額・重要度の閾値による経路差、承認却下、代行など）は明示ノードとして残す**。

## §⑤ 分岐健全性（decision well-formedness）の定義

`node_type:"decision"` のノードは、次を**すべて**満たして初めて「分岐」と認める。`scripts/lib/flows.mjs` の `checkFlowStructure` が自動検査する（`scripts/verify_outputs.mjs` から呼ばれる）。

1. **出口が2本以上** — `branches.length >= 2`。1本なら分岐ではない（process誤分類か、片側の枝＝差戻し先・例外経路の取りこぼし）。→ **error**
2. **行き先が2経路以上** — `branches[].target` がユニークに2つ以上。全枝が同一ノードを指す“偽分岐”を弾く。→ **error**
3. **各枝にラベル** — `branches[].label`（「一致／不一致」等）が空でない。→ warning
4. **条件がある** — `condition` 必須。→ **error**

逆方向の取りこぼし検知:
- **非decisionが複数経路に分岐している** — `node_type !== "decision"` なのに2つ以上の異なる `target` へ分岐＝菱形にし忘れ。→ warning（単一の routing 枝、例外ノードが本流へ戻る等は対象外）
- **差戻し過多** — `edge_type:"return"` のエッジ数が判断ノード数を上回る＝逐次チェックの疑い。§3-1の集約を検討。→ soft warning

> 1本枝のdecisionは**自動格下げせずerrorで停止**する。1本しかないのは「もう片方の枝＝差戻し先や例外経路を書き忘れた」ことが大半で、自動格下げは取りこぼしを隠すため。

## 自己レビュー（生成後に必ず通す）

①〜⑥を満たすか確認し、特に⑤（分岐健全性）と§3（差戻しの集約）に違反していたら描き直す。検証は次で機械的に確認できる（`asis_flows_YYYYMMDD.json` は `render_outputs.mjs` 実行時に `analysis` へマージされ検証される）。

```bash
node scripts/render_outputs.mjs --date YYYYMMDD
node scripts/verify_outputs.mjs --date YYYYMMDD
```
