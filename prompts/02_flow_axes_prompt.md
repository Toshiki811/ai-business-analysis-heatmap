# 02 フロー軸・業務カテゴリー抽出プロンプト

## 目的

`input/normalized/` の正規化済みドキュメントから、ヒートマップの横軸となる業務フローと、縦軸となる業務カテゴリーを抽出する。

この工程ではスコアリング、TOP3選定、HTML生成は行わない。

## 入力

- `input/normalized/` 内の全 `.md` / `.txt` ファイル
- 既に作成済みの `output/matrix_YYYYMMDD.md` がある場合は参考にしてよい

## 出力

`output/flow_axes_YYYYMMDD.json`

```json
{
  "metadata": {
    "created_at": "YYYY-MM-DD",
    "source_documents": ["ファイル名"]
  },
  "flow_columns": [
    {
      "group": "大分類フロー名",
      "steps": ["具体ステップ1", "具体ステップ2"]
    }
  ],
  "categories": ["業務分類1", "業務分類2"],
  "normalization_notes": [
    {
      "flow_step": "共通ステップ名",
      "source_task_examples": ["元タスク名1", "元タスク名2"],
      "reason": "正規化した理由"
    }
  ]
}
```

## 抽出ルール

- 大分類フローは業務全体を5〜8フェーズに整理する。
- 各大分類には1〜3個の具体ステップを置き、`flow_columns[].steps[]` の合計は原則10個にする。文書実態により8〜12個までは許容する。
- 具体ステップ名は10〜15文字を目安にし、後続の `heatmap_cells[].flow_step` と完全一致できる名称にする。
- 業務カテゴリーは入力文書の実態に合わせて5〜8個抽出する。
- 横軸ステップは個別タスク名をそのまま並べず、最大公約数の共通ステップ名に正規化する。
- 業務分類に該当しないステップの組み合わせは、この工程では作らない。

## 共通ステップ名の例

| 共通ステップ名 | 対象タスク例 |
|---|---|
| 受付 | 問合せ受付、申請受付、依頼受付 |
| 内容確認 | 内容確認、内容チェック、情報確認 |
| 情報収集・照合 | 情報調査、情報検索、資料確認、突合 |
| 文書・帳票作成 | 回答作成、文書作成、書類作成 |
| 承認・確認 | 上長承認、最終確認、承認依頼 |
| システム入力 | CRM入力、システム登録、会計入力 |
| 送信・連絡 | 顧客送信、連絡、通知 |
| 記録・報告 | 対応記録、月次レポート、報告書作成 |

## 注意事項

- この工程の出力を、後続の `05_score_prompt.md` と `06_top3_flow_prompt.md` の正本として使う。
- `flow_columns[].steps[]` は、業務整理マトリクス、クライアント入力CSV、ヒートマップで共通利用する横軸の正本である。
- `flow_columns[].steps[]` に存在しない文字列を後続工程の `flow_step` に使わない。
- 判断に迷う場合は `normalization_notes` に根拠を残す。
