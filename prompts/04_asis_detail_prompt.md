# 04 詳細As-Isフロー抽出プロンプト

## 目的

`input/normalized/` のマニュアル原文と `output/matrix_YYYYMMDD.md` をもとに、業務種別ごとの詳細As-Isフロー（分岐・差戻し・例外処理・実担当者・入出力帳票を含む）を抽出する。

この工程の出力は `templates/app/js/flow_svg.js` がブラウザ内でSVGスイムレーン図（動的レーン、判断ひし形、書類シンボル、推定ノードの破線、差戻しの赤点線、凡例）へ描画する。draw.io XMLは書かない。

この工程ではスコアリング、TOP3選定、CSV生成、HTML生成は行わない。

## 最優先方針

**目的＝AI・自動化による改善余地の特定。網羅性より可読性を優先し、人手の「判断・転記・照合・ハンドオフ」がどこにどれだけあるかが一目で読めること。** 起こりうる例外を全部描くのではなく、改善検討に効く構造を読ませる。粒度・分岐・例外の判断はすべてこの方針から導く。詳細な評価軸と作図ルールは [`docs/asis_flow_guideline.md`](../docs/asis_flow_guideline.md) を正本とする。

## 入力

- `input/normalized/` 内の全 `.md` / `.txt` ファイル（分岐・例外・担当者の根拠の正本）
- `output/matrix_YYYYMMDD.md`（タスク紐づけの正本）
- `output/flow_axes_YYYYMMDD.json`
- `input/source/client_input_filled.csv` または `input/source/client_input_filled_YYYYMMDD.csv`（存在する場合。回答済み分岐の反映に使う）

## 粒度ルール（分割トリガーで決める）

ノード数の絶対値ではなく、**何が起きたら割るか／割らないか**で粒度を決める（縦串＝隣接ノードの抽象度を揃える）。

- 1ノード = 1主体の1アクション（起票する、照合する、承認する、入力する）。`受付・起案` `内容確認` のような抽象工程名は禁止。マニュアルの手順記述の単位に合わせ、`請求書と納品書の照合` `支払伝票の会計システム入力` のように対象と動作が分かる名称にする。
- **割る（＝別ノードにする）トリガー**（いずれか該当で別ノード）: (1) 主体が変わる（ハンドオフ） (2) 使うシステム・ツールが変わる（会計システム→Excel→紙） (3) 待ち・承認・受け渡しが発生する (4) 判断（経路が分岐する）が入る (5) 媒体が変わる（紙起票→システム入力）。
- **割らない（＝1ノードにまとめる）条件**: 同一主体が同一ツールで連続して行い、途中に判断もハンドオフもない作業はまとめる。過分割も可読性を損なう。
- 分解チェックリスト: 各タスクについて次の観点を別ノードに分解できないか必ず確認する。(1) 資料・依頼の受領 (2) 転記・記入 (3) 照合・突合 (4) システム入力・記帳 (5) 承認依頼と承認実施（主体が異なれば必ず別ノード） (6) 帳票の出力・保管 (7) 関係者への連絡・送付
- 目安として業務種別あたり概ね10〜25ノード、6以下は粗すぎ（その場合はマニュアルの手順記載を再確認）。最低でも `当該業務種別のマトリクスタスク行数 × 1.5 + 2（start/end）` を下回らないこと。ただし上記トリガーが優先（数合わせで中身を雑にしない）。
- 各業務種別の先頭に `start`、末尾に `end` ノードを置く。

## actor・スイムレーンのルール

- `actor` にはマニュアル原文の役職名・部門名・システム名をそのまま使う（例: `理事長`、`会計責任者`、`出納職員`、`事務職員`、`会計システム`、`取引先`）。
- `担当者・業務部門` のような固定テンプレートレーンに寄せない。登場する実主体をすべて `actors[]` に登場順で列挙する。スイムレーンは `actors[]` から動的に生成される。
- 同一人物を指す表記ゆれ（`経理担当` と `経理担当者` 等）は1つに統一する。

## 分岐・差戻し・例外のルール

- マニュアルの「〜の場合」「不一致のとき」「承認されないとき」「期限を過ぎたとき」などの条件記述は、必ず `node_type: "decision"` + `condition` + `branches` で表現する。
- `branches[].target` は同一フロー内の任意のノードidを指せる。
  - 前方のノードへ戻る差戻し・再提出: `edge_type: "return"`（赤点線の戻り矢印として描画される）
  - 別系統の処理へ合流・スキップする前方遷移: `edge_type` 省略または `"normal"`
  - 例外処理ノードへの遷移: `edge_type: "exception"`（オレンジ点線として描画される）
- 例外対応の作業（取引先への照会、修正依頼、通常手続への切替など）は `node_type: "exception"` のノードにする。
- 承認ノードの直後に機械的に「承認可否」分岐を置かない。差戻しの分岐を置くのは、(a) マニュアルに差戻し・却下時の記述がある場合（explicit）、または (b) 推定分岐の6類型に該当する場合（inferred、後述）のみ。

### 検証はゲートに集約する（最重要）

- **入力1つごとに差戻しチェックを挟まない。** 一連の入力をまとめた後に1つのレビュー/承認ゲート（decision）を置き、不備は一括で差戻す。「早期に弾かれれば後段不要」はゲート集約で構造的に解消する。
- **差戻しエッジ（`return`）は状態が後退するものだけ。** (a) 他主体へ差し戻す（レーンをまたぐ） か (b) 前工程へ戻る もののみ。その場で直して進む「自己修正」は分岐にせず、ノードの `description` に書く。
- decisionは**経路が実際に変わる**分岐だけにする。95%通過する品質チェックは分岐にしない（結果が同じなら描かない）。

### 分岐健全性（decision well-formedness）

`node_type: "decision"` のノードは次をすべて満たす（`scripts/verify_outputs.mjs` の `checkFlowStructure` が自動検査し、違反は **error で停止**）。正本は [`docs/asis_flow_guideline.md`](../docs/asis_flow_guideline.md) §⑤。

- **出口2本以上**（`branches.length >= 2`）。1本は分岐ではない＝process誤分類か片側枝（差戻し先・例外経路）の取りこぼし。
- **行き先2経路以上**（`branches[].target` がユニークに2つ以上）。全枝が同一ノードを指す偽分岐は不可。
- **各枝にラベル**（`label`）と **`condition`** 必須。
- 逆に、非decisionノードが2つ以上の異なる `target` へ分岐していたら菱形にし忘れ（warning）。`node_type: "decision"` にする。

## 入出力帳票のルール

- ノードが扱う帳票・資料・データを `inputs[]`（参照・受領するもの）と `outputs[]`（作成・更新するもの）に記録する。
- 書類名はマニュアルに登場する名称をそのまま使う（例: `支払稟議書`、`小口現金出納帳`、`振込控`）。マニュアルにない書類名を発明しない。
- フロー図では書類シンボルとしてノードの近傍に描画される。すべてのノードに書く必要はなく、フロー理解に役立つ主要な帳票だけでよい。

## ハルシネーション抑止ルール（最重要）

- `confidence: "explicit"` のノード・分岐には `source_quote`（マニュアル原文の引用、40字以内）を必ず付ける。引用できない分岐は explicit にしない。
- 実務上ほぼ確実に存在するがマニュアルに記載がない分岐は、`confidence: "inferred"` として点線描画されるノードにし、判断ノードの場合は `hearing_item`（確認質問）を必ず付ける。
- 推定で作ってよい inferred 分岐は次の6類型に限定する:
  1. 承認 / 差戻し
  2. 照合一致 / 不一致
  3. 期限内 / 期限超過
  4. 資料不備・記載漏れあり / なし（不備時の差戻し先・照会先）
  5. 金額・重要度の閾値による経路の違い（承認者・決裁ルートが変わる等）
  6. 担当者不在時の代行 / 後回し処理
- 上記以外の推測はノード化せず、`unmodeled_notes` に「フロー化しなかった理由」とともに残す。
- `hearing_items[]` には inferred 分岐に対応する確認質問をまとめ、`node_id` で対象ノードを指す。これらは後続工程で業務整理マトリクスの確認事項・クライアント入力CSVへ自動転記される。

## 実態確認質問（reality_check）のルール

- inferred 分岐に紐づかない質問でも、「マニュアル記載と実態の乖離」を確認する質問は `hearing_items[]` に追加してよい。対象の `node_id` は該当する process ノードを指す。
- 各 hearing_item に `question_type` を付ける: 分岐確認は `"branch_confirmation"`、実態確認は `"reality_check"`（省略時は branch_confirmation として扱う）。
- reality_check の定型観点:
  - 実際の実施者はマニュアル記載の役職どおりか（兼務・代行の実態）
  - 実際にこの順序で処理しているか（複数件のまとめ処理・後回しの有無）
  - システム外で Excel・紙・手書き台帳による作業が挟まっていないか
  - マニュアルにない承認・確認ステップが実際には存在しないか
- `hearing_items` は業務種別あたり最低3件（うち reality_check を最低1件）、5件程度を推奨する。質問はAs-Isフローの配置・順序・分岐・レーンを変えうるものに限定し、定量系（頻度・件数・所要時間）は引き続き禁止する。

## タスク紐づけルール

- `process` / `exception` ノードは `source_task`（業務分類・業務種別・タスク順・タスク名の4項目、`output/matrix_YYYYMMDD.md` と完全一致）で業務整理マトリクスのタスクと紐づける。
- 1タスクの複数ノードへの分割は原則必須とする。タスクの `業務内容詳細` に動作（動詞）が2つ以上含まれる場合（「確認して入力する」「照合のうえ承認を得る」等）は必ずノードを分割する。逆に複数タスクを1ノードへ統合するのは禁止（ヒートマップ集計が壊れるため）。
- `decision` ノードや `start` / `end` には `source_task` を付けなくてよい。確認質問は直前の作業ノードのタスクへ自動で紐づく。
- `category` / `business_type` はマトリクスの `業務分類` / `業務種別` と完全一致させる。存在しない組み合わせを作らない。

## 再実行時（ヒアリング後）のルール

- クライアント回答で分岐・差戻し・例外が確定した場合は、該当ノードの `confidence` を `explicit` に昇格し、`source_quote` の代わりに `client_answer` フィールドへ回答内容を記録、`hearing_item` を除去する。
- `（確認不要）` と回答された推定分岐は削除し、`unmodeled_notes` に「クライアント確認の結果、分岐なし」と残す。
- 回答済みの質問は `hearing_items[]` に再掲しない。

## 自己レビュー（出力前に必ず通す）

各業務種別のフローについて以下を確認し、違反があれば描き直す（評価軸の正本は [`docs/asis_flow_guideline.md`](../docs/asis_flow_guideline.md) §1）:

1. 正常系を start→end まで主線1本で指でなぞれるか。
2. 隣接ノードの抽象度が揃い、§粒度の分割トリガーに沿っているか（抽象工程名が残っていないか）。
3. 主体が変わる点でレーンをまたいでいるか（ハンドオフが見えるか）。
4. 手作業/システム/判断の境界でノードが割れているか。
5. **すべての decision が出口2本以上・行き先2経路以上・各枝ラベルあり・condition あり**か。検証はゲートに集約され、差戻しが逐次に乱立していないか。
6. Excel・紙・口頭などシステム外作業が明示されているか。

## 検証

出力後、以下で契約を確認できる（`asis_flows_YYYYMMDD.json` は次回の `render_outputs.mjs` 実行時に `analysis` へマージされ検証される）。

```bash
node scripts/render_outputs.mjs --date YYYYMMDD
node scripts/verify_outputs.mjs --date YYYYMMDD
```

- ノードidの一意性、`branches[].target` / `next` の解決可能性、decisionノードの分岐健全性（出口2本以上・2経路以上・condition必須）、inferred decisionの `hearing_item` 必須が機械検証される（`scripts/lib/flows.mjs` の `checkFlowStructure`）。
- `category` / `business_type` が `matrix_tasks` に存在しない場合はエラーになる。
- 単体テスト: `node --test scripts/test/flow_structure.test.mjs`。

## 注意事項

- 業務種別ごとに1フローを作る。`flows[]` は `output/matrix_YYYYMMDD.md` の全業務種別を網羅すること（スキップ禁止）。手順記述がマニュアルに薄い業務種別は、`matrix_tasks` のタスク行・`業務内容詳細` を骨格として分解チェックリストで細分化し、推測箇所は `inferred` + `hearing_item` で明示する。フロー化できない部分のみ `unmodeled_notes` に残す。詳細フローのない業務種別は直列タスクフローで描画されてしまい、粒度フィードバックの再発要因になる。
- draw.io XMLはこの工程では作らない。描画はブラウザ内で `templates/app/js/flow_svg.js`（自前インラインSVGレンダラ）が行う（動的スイムレーン、判断ひし形、書類シンボル、推定ノードの破線、差戻しの赤点線、凡例を自動生成する）。
- 後続の `06_score_prompt.md` は、本工程の decision ノード（explicit な分岐条件）を「ルール化可能シグナル」として自動化手段の判定に使う。
